toHTML PDF Readable HTML Specification

This document defines the PDF support profile for toHTML: enough compliance with ISO 32000 PDF standards to turn a readable, selectable-text PDF into readable, structured HTML.

This is an implementation specification for the converter. It is not a claim of full PDF reader conformance and it is not a pixel-perfect rendering specification.

1. Source Standards

The implementation profile is derived from the two local standards files in the repository:

  • PDF32000_2008.pdf: ISO 32000-1:2008, PDF 1.7.
  • ISO_32000-2_sponsored-ec2.pdf: ISO 32000-2:2020, PDF 2.0, including Errata Collection 2 annotations available in the local copy.

New implementation work should use ISO 32000-2 as the primary reference where it clarifies or updates ISO 32000-1. ISO 32000-1 remains necessary for legacy PDF 1.7 behavior, especially older files that rely on features deprecated or clarified in PDF 2.0.

Standard text is intentionally paraphrased here. This document identifies clauses and implementation obligations without copying the standards.

2. PDF Readable HTML Profile

The profile is goal-based. Earlier goals create the minimum reliable extraction path. Later goals add semantic structure and formatting fidelity. Each goal is independently testable.

2.1 Requirement Terms

  • MUST: required for the goal to be considered implemented.
  • SHOULD: expected unless there is a documented reason not to do it.
  • MAY: optional behavior that can improve output without changing the compliance baseline.
  • OUT OF SCOPE: deliberately excluded from this profile unless a later project stage reopens it.

2.2 Definition: Readable PDF

For this application, a readable PDF is a PDF whose meaningful document content is available through PDF text-showing operations, tagged PDF structure, replacement text, or other standard PDF text mechanisms.

A scanned page image without selectable text is not a readable PDF for this profile. The converter MUST report such pages as non-extractable placeholders unless OCR is explicitly added as a separate feature.

2.3 Definition: Readable HTML

Readable HTML means semantic HTML that preserves the document's textual content, reading order, page boundaries when useful, headings, paragraphs, lists, tables, links, figures, and extraction warnings. It does not mean exact font, color, coordinate, or page-layout reproduction.

Generated converter output MUST remain aligned with the application contract: no CSS and no JavaScript by default. Structure comes from HTML elements and attributes.

4. Implementation Goals

Goal 0: Define the Supported PDF Surface

Establish an honest support contract before parsing begins.

Relevant standard areas

  • ISO 32000-1 and ISO 32000-2: scope and conformance terminology.
  • ISO 32000-2: terminology changes replacing the older broad "conforming reader" framing with processor-specific expectations.

Implementation requirements

  • The converter MUST describe itself as a best-effort PDF-to-HTML extractor, not as a full PDF renderer or complete conforming PDF reader.
  • The converter MUST support unencrypted, local PDF files with selectable text as the primary target.
  • The converter MUST preserve readable content over visual fidelity.
  • The converter MUST produce warnings for unsupported but detected features that may affect content extraction.
  • The converter MUST NOT silently OCR scanned pages.
  • The converter SHOULD expose a machine-readable warning list in the internal document model and a human-visible warning area in generated HTML when warnings exist.

Acceptance criteria

  • A PDF with no extractable text produces page placeholders and a warning.
  • A PDF with unsupported filters or encryption produces an explicit error or warning rather than corrupted text.
  • Documentation states exactly which PDF features are supported at the current goal level.

Goal 1: Parse PDF Syntax Without False Text Extraction

Replace raw byte scanning with a real PDF object parser for the subset needed by readable extraction.

Relevant standard areas

  • 7.2 Lexical conventions: white space, delimiters, comments, tokens.
  • 7.3 Objects: booleans, numbers, strings, names, arrays, dictionaries, streams, null, indirect objects.
  • 7.3.4 String objects: literal strings, escapes, balanced parentheses, hexadecimal strings.
  • 7.3.8 Stream objects: stream dictionary, stream extent, line endings around stream data.

Implementation requirements

  • The parser MUST tokenize PDF lexical syntax according to standard delimiter and white-space rules.
  • The parser MUST parse literal strings with nested parentheses and escape sequences.
  • The parser MUST parse hexadecimal strings, including odd digit count handling.
  • The parser MUST parse name objects, including escaped name bytes introduced with # hexadecimal escapes.
  • The parser MUST parse arrays and dictionaries recursively enough to resolve page trees, resources, fonts, content streams, CMaps, and structure trees.
  • The parser MUST parse indirect references in the form object generation R, not only generation 0.
  • The parser MUST treat stream bytes as bytes, not as text, until the stream dictionary is parsed and filters are applied.
  • The parser MUST NOT search all parenthesized byte ranges as document text.
  • The parser SHOULD preserve byte offsets for diagnostics and source spans.

Acceptance criteria

  • Text in image streams, font programs, metadata, and compressed binary streams is not emitted as document text.
  • PDF strings with escaped parentheses and octal escapes extract correctly.
  • Indirect objects with non-zero generations can be resolved when referenced by the current xref data.

Goal 2: Locate the Current Document Objects

Use the PDF file structure so extraction operates on the current object graph rather than every byte sequence that looks interesting.

Relevant standard areas

  • 7.5 File structure: header, body, cross-reference data, trailer.
  • 7.5.4 Cross-reference tables.
  • 7.5.5 File trailer and root catalog.
  • 7.5.6 Incremental updates.
  • 7.5.7 Object streams.
  • 7.5.8 Cross-reference streams.

Implementation requirements

  • The parser MUST locate the final startxref entry and follow the active cross-reference chain.
  • The parser MUST use the latest revision of each object according to cross-reference data and incremental update rules.
  • The parser MUST read the trailer dictionary and resolve the document catalog through /Root.
  • The parser MUST support classic cross-reference tables for PDF 1.x files.
  • The parser SHOULD support cross-reference streams because they are common in modern PDFs.
  • The parser SHOULD support object streams because modern PDFs may compress ordinary objects into streams.
  • If xref streams or object streams are not yet supported, the converter MUST emit a clear warning or error when content depends on them.
  • The parser MUST avoid choosing stale object revisions from earlier incremental updates.

Acceptance criteria

  • A PDF with an incremental update extracts the updated text, not the original stale text.
  • A PDF using a classic xref table resolves pages, resources, fonts, and content streams through indirect references.
  • A PDF using an unsupported xref stream or object stream fails with an actionable diagnostic rather than partial nonsense output.

Goal 3: Decode Content Streams Reliably

Decode the stream types required to reach page text, CMaps, resources, and nested form content.

Relevant standard areas

  • 7.3.8 Stream objects.
  • 7.4 Filters.
  • 7.4.2 ASCIIHexDecode.
  • 7.4.3 ASCII85Decode.
  • 7.4.4 LZWDecode and FlateDecode.
  • 7.4.5 RunLengthDecode.
  • 7.4.10 Crypt filter.

Implementation requirements

  • The stream decoder MUST respect /Length and stream extent rules.
  • The stream decoder MUST support unfiltered streams.
  • The stream decoder MUST support /FlateDecode for page content streams and CMaps.
  • The stream decoder MUST support filter arrays and apply filters in order.
  • The stream decoder SHOULD support /ASCIIHexDecode, /ASCII85Decode, and /RunLengthDecode for content streams.
  • The stream decoder SHOULD support /LZWDecode when encountered in legacy PDFs.
  • The stream decoder MAY ignore image-only filters such as /DCTDecode, /JPXDecode, /CCITTFaxDecode, and /JBIG2Decode for OCR purposes, but MUST still identify the stream as image content when possible.
  • The stream decoder MUST respect /DecodeParms when those parameters affect decoded content required for extraction.
  • The stream decoder MUST produce a warning when an unsupported filter prevents extraction of page text, CMaps, object streams, or forms.

Acceptance criteria

  • A Flate-compressed page content stream extracts the same text as its uncompressed equivalent.
  • A stream with multiple filters decodes in the correct order.
  • A stream with an unsupported text-critical filter records which filter blocked extraction.

Goal 4: Walk Pages in Document Order

Extract from page objects rather than arbitrary streams.

Relevant standard areas

  • 7.7 Document structure.
  • 7.7.2 Document catalog.
  • 7.7.3 Page tree, page tree nodes, page objects, inherited page attributes.
  • 7.8 Content streams and resources.

Implementation requirements

  • The converter MUST resolve the catalog /Pages tree.
  • The converter MUST traverse the page tree in page order.
  • The converter MUST extract only from page content streams and nested Form XObjects reachable from pages.
  • The converter MUST support page /Contents as a single stream or an array of streams.
  • The converter MUST apply inherited page attributes relevant to extraction, including /Resources, /MediaBox, /CropBox, and /Rotate where present.
  • The converter SHOULD create a PageBreak or page source boundary between extracted pages.
  • The converter SHOULD attach page numbers to source spans, warnings, placeholders, and extracted blocks.

Acceptance criteria

  • Output text follows page-tree order, not object-number order.
  • Pages with inherited resources can resolve fonts and CMaps.
  • Empty or non-extractable pages create page placeholders without collapsing neighboring page text.

Goal 5: Interpret Text Objects and Text-Showing Operators

Convert PDF text instructions into positioned text segments.

Relevant standard areas

  • 9.3 Text state parameters and operators.
  • 9.4 Text objects.
  • 9.4.2 Text-positioning operators.
  • 9.4.3 Text-showing operators.
  • 9.4.4 Text space details.

Implementation requirements

  • The content stream interpreter MUST recognize BT and ET text object boundaries.
  • The interpreter MUST maintain the text matrix and text line matrix.
  • The interpreter MUST maintain relevant text state: character spacing, word spacing, horizontal scaling, leading, font, font size, text rendering mode, and text rise.
  • The interpreter MUST support text-positioning operators Td, TD, Tm, and T*.
  • The interpreter MUST support text-showing operators Tj, TJ, apostrophe, and double quote.
  • The interpreter MUST interpret TJ numeric adjustments as positioning changes that can indicate word gaps or tight kerning.
  • The interpreter MUST advance text position using font metrics when available.
  • The interpreter SHOULD track graphics state save and restore with q and Q where it affects the current transformation matrix.
  • The interpreter SHOULD account for page rotation and current transformation matrix enough to produce correct reading order for common rotated pages.
  • The interpreter MUST ignore text rendering modes only for visual styling, not for whether text exists. Invisible or clipping text requires Goal 13 handling.

Acceptance criteria

  • Simple PDFs using Tj, TJ, Td, TD, and Tm extract words in the expected order.
  • Reasonable word spacing is preserved for text split across multiple show operations.
  • Line breaks are inferred from text matrices and line movement, not from stream byte layout.

Goal 6: Map PDF Character Codes to Unicode

Produce real text rather than font-specific glyph codes.

Relevant standard areas

  • 9.6 Simple fonts.
  • 9.6.5 Character encoding in PDF 2.0, corresponding 9.6.6 in PDF 1.7.
  • 9.7 Composite fonts and CMaps.
  • 9.10 Extraction of text content.
  • 9.10.3 ToUnicode CMaps.
  • Annex D Character sets and encodings.

Implementation requirements

  • The converter MUST use a font dictionary's /ToUnicode CMap when present.
  • The converter MUST parse ToUnicode mappings from beginbfchar and beginbfrange.
  • The converter MUST support one-byte and multi-byte source character codes according to CMap codespace ranges.
  • The converter MUST support ToUnicode targets containing multiple Unicode scalar values, such as ligatures mapped to multiple letters.
  • The converter MUST decode ToUnicode target strings as UTF-16BE unless the standard specifies another interpretation for the relevant string form.
  • The converter SHOULD support predefined encodings for simple fonts, including WinAnsiEncoding, MacRomanEncoding, MacExpertEncoding, StandardEncoding, Symbol, and ZapfDingbats where needed for fallback.
  • The converter SHOULD support Differences arrays for simple font encodings.
  • The converter SHOULD support Type 0 fonts with Identity-H and Identity-V CMaps when paired with ToUnicode or usable CID mapping.
  • The converter MUST record a warning when text cannot be mapped to Unicode with acceptable confidence.
  • The converter MUST NOT emit private glyph codes or raw binary as if they were readable text.

Acceptance criteria

  • ASCII, WinAnsi, and ToUnicode-based PDFs extract expected Unicode text.
  • Ligature mappings such as a single glyph representing "fi" emit "fi" when ToUnicode provides that mapping.
  • Multi-byte CMaps do not split character codes into incorrect single-byte fallback characters.

Goal 7: Reconstruct Lines, Words, and Paragraphs

Convert positioned glyph runs into readable prose.

Relevant standard areas

  • 9.3 Text state and spacing parameters.
  • 9.4 Text positioning and text space.
  • 9.10 Text extraction.
  • 14.8.2.6 in PDF 2.0 and 14.8.2.5 in PDF 1.7 for identifying word breaks in tagged PDF contexts.

Implementation requirements

  • The layout pass MUST group text segments into lines using transformed baseline positions and font size tolerance.
  • The layout pass MUST order text within a line by writing direction and x position for common left-to-right horizontal text.
  • The layout pass SHOULD support right-to-left and vertical writing as detected by font/CMap or text direction signals, but MAY initially warn when unsupported.
  • The layout pass MUST infer word gaps from actual segment positions, glyph advances, word spacing, and TJ adjustments.
  • The layout pass MUST avoid inserting spaces inside words merely because a word was split across text-showing operations.
  • The layout pass SHOULD join lines into paragraphs based on indentation, vertical gap, font size, and consistent alignment.
  • The layout pass SHOULD preserve hard line breaks only when the source layout indicates a list item, address, poem, code-like text, table row, or intentionally separate line.
  • The layout pass MUST normalize repeated extraction whitespace into readable text while preserving meaningful spaces.
  • The layout pass SHOULD dehyphenate line-ending soft hyphenation when evidence indicates a word was split only because of line wrapping.

Acceptance criteria

  • Words do not collapse into strings such as "withoutChangingthe" when measurable gaps exist.
  • Words do not split into strings such as "programm er" when glyph metrics indicate continuous text.
  • Multi-line paragraphs become single HTML paragraphs unless layout indicates separate blocks.

Goal 8: Detect Basic Document Blocks

Infer headings, paragraphs, lists, and block boundaries from layout when no tagged structure is available.

Relevant standard areas

  • 9.3 and 9.4 for text size, position, and baseline information.
  • 14.8.4 for standard structure types when tags are available.

Implementation requirements

  • The converter MUST emit paragraph blocks for ordinary prose.
  • The converter SHOULD infer headings from a combination of larger font size, bold font naming cues, isolated line position, vertical spacing, numbering patterns, and repeated document conventions.
  • The converter SHOULD assign heading levels consistently within a document, using relative hierarchy rather than absolute font size alone.
  • The converter SHOULD detect unordered list items from bullet glyphs and indentation.
  • The converter SHOULD detect ordered list items from numbering patterns and indentation.
  • The converter SHOULD keep continuation lines inside the same list item when indentation and vertical spacing indicate continuation.
  • The converter MUST avoid treating page headers, footers, and page numbers as main content when repeated-region detection identifies them.
  • The converter SHOULD preserve source page numbers in block source spans.

Acceptance criteria

  • A basic report with large section titles emits h1 through h6 where hierarchy is inferable.
  • Bullet and numbered lists become ul and ol with nested li content.
  • Repeated headers and footers do not pollute the main text stream by default.

Goal 9: Reconstruct Tables from Selectable Text

Produce semantic HTML tables when a table is represented by selectable text and positional layout, even if the PDF has no table tags.

Relevant standard areas

  • 9.3 and 9.4 for text positioning and spacing.
  • 14.8.4 table structure types when tagged PDF is available.
  • 14.8.5 table-related layout attributes when present.

Implementation requirements

  • The table detector MUST operate only on selectable text unless OCR is later added.
  • The table detector MUST cluster candidate rows by baseline position.
  • The table detector MUST cluster columns by x alignment, repeated gaps, ruling-line hints if available, and consistent cell boundaries across adjacent rows.
  • The table detector SHOULD use vector drawing lines and rectangles as hints for cell boundaries, but MUST NOT require ruling lines to detect a table.
  • The table detector SHOULD use font weight, background rectangles, top row repetition, and tag data to infer header cells.
  • The table detector SHOULD support cells spanning multiple columns or rows when text regions or ruling geometry indicate spans.
  • The table detector SHOULD distinguish multi-column page layout from data tables by checking row alignment repetition, column count stability, header patterns, and inter-column prose flow.
  • The converter MUST emit table, thead when header rows are known, tbody, tr, th, and td as appropriate.
  • The converter MUST preserve cell reading order and inline text inside cells.
  • The converter MUST emit a warning when a likely table is image-only or otherwise non-extractable.

Acceptance criteria

  • A simple selectable-text table with aligned columns becomes an HTML table.
  • A table without visible grid lines but with repeated aligned columns becomes an HTML table.
  • Two-column prose is not incorrectly converted into a table.
  • Image-only specification tables produce a warning rather than an empty or fabricated table.

Goal 10: Extract Links, Images, Figures, and Captions

Preserve non-textual content that contributes to document readability.

Relevant standard areas

  • 8.8 External objects and XObjects.
  • 8.9 Images and inline images.
  • 8.10 Form XObjects.
  • 12.5 Annotations, especially link annotations.
  • 14.8.4 figure and caption structure types.
  • 14.9.3 Alternate descriptions.

Implementation requirements

  • The converter SHOULD detect image XObjects and inline images reachable from page content streams.
  • The converter MAY extract image assets to the configured asset directory when image extraction is enabled.
  • The converter MUST warn when a page contains image content that may contain non-selectable text and OCR is not enabled.
  • The converter SHOULD detect link annotations and attach URLs to overlapping text ranges as HTML anchors.
  • The converter SHOULD use figure tags, captions, and alternate descriptions from tagged PDF when available.
  • The converter SHOULD use /Alt text for image alt text when provided by the structure tree.
  • The converter MAY create figure and figcaption output where figure-caption association is reliable.
  • The converter MUST NOT invent alt text for images without evidence; it should leave alt empty or record a warning depending on output policy.

Acceptance criteria

  • A PDF link annotation over visible text becomes an HTML a element with the correct href.
  • A tagged figure with alternate text maps to an HTML image or figure with appropriate alt text.
  • A scanned image page produces a non-extractable placeholder and image/OCR warning.

Goal 11: Use Tagged PDF Structure When Available

Prefer author-provided logical structure over geometric inference.

Relevant standard areas

  • 14.6 Marked content.
  • 14.7 Logical structure.
  • 14.8 Tagged PDF.
  • 14.8.4 Standard structure types.
  • 14.8.5 Standard structure attributes.
  • 14.9 Repurposing and accessibility support.

Implementation requirements

  • The converter SHOULD detect the document catalog's /StructTreeRoot.
  • The converter SHOULD walk the logical structure tree in structure order when it is complete enough for extraction.
  • The converter SHOULD map standard structure types to internal document blocks: headings, paragraphs, lists, list labels, list bodies, tables, table rows, table headers, table cells, figures, captions, formulas, and links.
  • The converter MUST use marked-content IDs to associate structure elements with page content.
  • The converter MUST ignore artifacts as main document content when tags identify them as artifacts.
  • The converter SHOULD support role maps and namespaces, especially in PDF 2.0 where standard structure namespaces are more explicit.
  • The converter SHOULD use structure attributes for table headers, scope, row spans, column spans, and layout hints when present.
  • The converter SHOULD fall back to geometric extraction for untagged or incomplete portions of the document.
  • The converter MUST record a warning when tags are present but internally inconsistent enough to prevent reliable extraction.

Acceptance criteria

  • A tagged PDF with heading and paragraph tags produces corresponding HTML headings and paragraphs even if visual font sizes are ambiguous.
  • A tagged PDF table produces semantic table markup with headers and cell spans where specified.
  • Tagged artifacts such as page numbers and running headers are excluded from main content.

Goal 12: Honor Replacement Text, Language, and Accessibility Metadata

Use PDF extraction aids designed for repurposing content.

Relevant standard areas

  • 9.10 Text extraction.
  • 14.9.2 Natural language specification.
  • 14.9.3 Alternate descriptions.
  • 14.9.4 Replacement text.
  • 14.9.5 Expansion of abbreviations and acronyms.
  • 14.9.6 Pronunciation hints in PDF 2.0.

Implementation requirements

  • The converter MUST prefer /ActualText over visually shown glyph text for the marked-content sequence or structure element to which it applies.
  • The converter SHOULD use /Alt for alternate descriptions of figures and non-textual content.
  • The converter SHOULD preserve document language from the catalog /Lang and more specific structure language where available.
  • The converter SHOULD use expansion text for abbreviations when it improves readable output or can be represented semantically.
  • The converter MAY preserve pronunciation hints in metadata or warnings if there is no direct HTML representation in the current document model.
  • The converter MUST apply replacement text only to the smallest applicable content range to avoid duplicating surrounding text.

Acceptance criteria

  • A symbol or formula with /ActualText emits the replacement text instead of unreadable glyph names.
  • Document language is reflected in the generated HTML lang attribute when available.
  • Alternate text is preserved for tagged figures.

Goal 13: Handle Hidden, Invisible, and Duplicate Text

Avoid extracting content that harms readability, while preserving text deliberately supplied for accessibility or copy/paste.

Relevant standard areas

  • 9.3.6 Text rendering mode.
  • 14.8.2 Tagged PDF and page content.
  • 14.8.2.4 Hidden or invisible page content in PDF 2.0.
  • 14.8.2.5 Page content order and logical content order in PDF 2.0.

Implementation requirements

  • The converter SHOULD detect text rendering modes and graphics state that make text invisible or clipping-only.
  • The converter SHOULD exclude invisible duplicate text when it merely shadows visible text and harms readability.
  • The converter MUST preserve invisible text when tagged structure or replacement text indicates it is the intended accessible representation.
  • The converter SHOULD deduplicate overlapping identical text segments produced by rendering tricks or layered content.
  • The converter SHOULD use tagged logical order over paint order when both are available and disagree.

Acceptance criteria

  • PDFs with hidden OCR text over a scanned page do not emit repeated nonsense if the text is poorly aligned or non-readable; they warn when confidence is low.
  • PDFs with accessible replacement text emit the intended text once.
  • Layered duplicate visible text is not duplicated in HTML output.

Goal 14: Preserve Readable Formatting Without CSS

Map supported PDF semantics into the existing HTML document model using semantic elements only.

Relevant standard areas

  • 9.3 Text state for font size and styling signals.
  • 9.6 Fonts for bold, italic, symbolic, and width information.
  • 14.8 Standard structure types and attributes.

Implementation requirements

  • The converter MUST output semantic block elements: headings, paragraphs, lists, tables, images, block quotes when reliably detected, page breaks, and placeholders.
  • The converter SHOULD output inline emphasis and strong emphasis when font naming, font descriptor flags, or tagged semantics make it reliable.
  • The converter SHOULD output code-style inline or block content only when monospaced font and layout evidence strongly indicate code or preformatted content.
  • The converter SHOULD preserve line breaks inside preformatted or address-like content only when layout evidence supports it.
  • The converter MUST NOT use CSS positioning to recreate the PDF page.
  • The converter MUST NOT output JavaScript.
  • The converter MAY add data attributes or source metadata if they remain useful for diagnostics and do not become visual-layout dependencies.

Acceptance criteria

  • A readable PDF report becomes a readable HTML document when opened in a plain browser with default styles.
  • Headings, lists, tables, and links are represented by semantic HTML elements rather than positioned spans.
  • Loss of visual styling does not prevent understanding the extracted document.

Goal 15: Report Unsupported Content Precisely

Make extraction limits visible and actionable.

Relevant standard areas

  • 7.4 Filters for unsupported stream decoding.
  • 7.6 Encryption.
  • 8.9 Images for image-only content.
  • 9.10 Text extraction limits.

Implementation requirements

  • The converter MUST distinguish between "no text exists", "text exists but cannot be decoded", and "text may exist only in images".
  • The converter MUST include page number and feature name in warnings whenever known.
  • The converter MUST create page placeholders for pages with no extractable text.
  • The converter SHOULD produce one warning per distinct cause rather than flooding output with repeated low-level warnings.
  • The converter SHOULD include enough detail for tests and users to identify unsupported filters, encryption, missing CMaps, malformed content streams, or image-only tables.

Acceptance criteria

  • An encrypted PDF reports encryption as the blocker.
  • A PDF page containing only an image XObject reports non-extractable image content.
  • A page with unsupported CMap or font encoding reports text decoding uncertainty.

Goal 16: Build a PDF Fixture and Regression Suite

Verify every goal with small controlled fixtures before relying on real-world PDFs.

Relevant standard areas

  • All clauses referenced by implemented goals.

Implementation requirements

  • Each parser feature MUST have focused fixtures with minimal PDFs or extracted object/content snippets.
  • Each layout feature SHOULD have golden HTML tests using realistic page geometry.
  • Fixtures MUST include uncompressed and Flate-compressed streams.
  • Fixtures SHOULD include classic xref tables, incremental updates, xref streams, object streams, inherited page resources, Form XObjects, ToUnicode CMaps, simple encodings, multi-byte CMaps, headings, lists, tables, links, artifacts, ActualText, image-only pages, and unsupported filters.
  • Regression tests MUST assert warnings as part of output, not only successful text.
  • Real sample PDFs MAY be used for broad regression coverage, but minimal fixtures are required for precise behavior.

Acceptance criteria

  • A failing PDF extraction can be reduced to a small fixture before the fix is accepted.
  • Golden tests cover text, tables, layout, warnings, and tagged PDF behavior as those goals are implemented.
  • Quality review includes parser complexity and module boundaries because PDF parsing can easily become unmaintainable.

5. Internal Model and HTML Mapping

PDF extraction SHOULD target the existing shared document model before rendering HTML. The mapping below defines the intended semantic output.

PDF evidence Internal model HTML output
Document title metadata or tagged document title DocumentMetadata.title title, optional top-level heading only when source content supports it
Catalog or structure language DocumentMetadata.language html lang
Tagged heading or inferred heading line Block::Heading h1 through h6
Running text grouped by layout or paragraph tag Block::Paragraph p
Tagged list or detected list markers and indentation Block::List ul, ol, li
Tagged table or detected row/column grid Block::Table table, thead, tbody, tr, th, td
Image XObject with reliable extraction Block::Image or Inline::Image img, optionally inside figure
Link annotation or link tag over text Inline::Link a href
Page boundary Block::PageBreak semantic page marker without CSS or JavaScript
Non-extractable page Block::PagePlaceholder placeholder element with reason text
Unsupported feature, uncertain decoding, skipped content ConversionWarning warning section or comments according to renderer policy

6. Recommended Build Order

  1. Goal 0: lock the support contract and warning taxonomy.
  2. Goal 1: implement real lexical/object parsing.
  3. Goal 2: implement xref, trailer, object resolution, and page tree traversal.
  4. Goal 3: implement stream decoding with filter arrays and warnings.
  5. Goal 4: extract per page with inherited resources.
  6. Goal 5: interpret text objects into positioned text segments.
  7. Goal 6: improve Unicode mapping with robust font and CMap support.
  8. Goal 7: reconstruct words, lines, and paragraphs.
  9. Goal 8: infer headings and lists from layout.
  10. Goal 9: reconstruct selectable-text tables.
  11. Goal 10: add links, images, figures, and captions where reliable.
  12. Goal 11: prefer tagged PDF logical structure.
  13. Goal 12: apply ActualText, Alt, language, and accessibility metadata.
  14. Goal 13: handle hidden, invisible, duplicate, and artifact content.
  15. Goal 14: refine semantic formatting without CSS.
  16. Goal 15: make unsupported content reporting precise.
  17. Goal 16: keep fixture coverage growing with every goal.

7. Explicit Non-Goals

  • Pixel-perfect page rendering is OUT OF SCOPE.
  • CSS-positioned HTML recreation of PDF pages is OUT OF SCOPE.
  • JavaScript output is OUT OF SCOPE.
  • OCR is OUT OF SCOPE unless added as a separate, explicit feature.
  • Full interactive PDF behavior is OUT OF SCOPE, including forms execution, multimedia, 3D, JavaScript actions, and rich annotation behavior.
  • Digital signature validation is OUT OF SCOPE except for detecting that signature-related structures exist if they affect extraction warnings.
  • Exact color, typography, kerning, and visual design fidelity are OUT OF SCOPE except where metrics improve readable extraction.
  • Encrypted PDFs requiring passwords are OUT OF SCOPE for automatic extraction unless password handling is explicitly implemented.

8. Definition of Done for PDF Readable HTML

The PDF implementation is good enough for the application when a typical selectable-text PDF report, manual, article, or specification sheet can be converted into HTML that a person can read from top to bottom without consulting the PDF for basic comprehension.

At that point, the converter MUST reliably:

  • read pages in page-tree order;
  • extract Unicode text from common simple and composite fonts;
  • preserve words, line order, and paragraph flow;
  • represent headings, lists, and tables when reliably available;
  • preserve links and figure alt text when available;
  • use tagged PDF semantics when present;
  • exclude artifacts and repeated headers/footers when detectable;
  • create placeholders for non-extractable pages;
  • emit precise warnings for unsupported or uncertain content;
  • render semantic HTML with no CSS and no JavaScript by default.
