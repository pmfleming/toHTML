toHTML Output Alignment Plan

Plan date: 2026-05-23. Author: Claude (Opus 4.7). Scope: bring the converter's output closer to its input across the ten PDFs in input/ by comparing each input to the current HTML in output/. Each stage targets a specific failure class observed in those samples and finishes with an effectiveness check against the same corpus.

This plan is grounded in the current source tree under src/pdf/ and the present HTML output only. It does not reference the spec or earlier review docs.

Each output combines a positional reconstruction (<section class="pdf-recreated-page">) with a semantic article inside <details class="pdf-extracted-content">. The observations below focus on the semantic article, because that is what a downstream LLM, screen reader, or search index will use.

2. Failure classes ranked by impact across the corpus

  1. Glyph-to-Unicode failure when /ToUnicode is missing. Cripples HCE NDA entirely and most icon strings in DREU2021. Touches potentially any third-party-produced PDF. Highest impact.
  2. Title / metadata leaks. Several documents take their /Info /Title verbatim and end up with titles like "Microsoft Word - …", "IAF210906-1.xlsx", or "English". Cheap to fix and very visible.
  3. Multi-line title and multi-line heading consolidation. IEC cover, How To Program headings, Inventronics title bar. Currently every visually-stacked line is its own heading.
  4. List recovery from PDF marker glyphs. Bullets and numbered items in Inventronics, Installation Guidelines, How To Program become flat paragraphs.
  5. Reading order in multi-column and form layouts. IEC bilingual cover, Silica org chart, IAF-Dimitri form.
  6. TOC dot-leader detection. The dot-leader "Title ......... 3" rows become loose paragraphs or weird tables in IEC and XML-Message-for-SCT.
  7. Word-boundary noise from font width estimates. "Pow er", "default C urrent", "AC mA)" / "d driver" — a handful of mid-word splits even after the existing gap classifier.
  8. Visual reconstruction noise. The pdf-recreated-page emits hundreds of one-word spans and occasional ghost shapes; collapses to look nothing like the source.
  9. Page-furniture pollution. The "English" watermark in IEC, the trailing "1" footnote marker above the SCT table, and repeated headers on long docs still appear as content.
  10. Spurious heading classification. Installation Guidelines and Inventronics turn enlarged callout fragments into <h2>.

3. Staged plan

Stages are ordered by impact-per-effort against the failure classes above. Every stage ends with a concrete effectiveness check that re-runs the converter on the corpus and reports the delta. Each check belongs in the stage that introduces the change so any regression on another sample is caught immediately.

Stage 1 — Recover Unicode from fonts without a /ToUnicode CMap

The single biggest content failure in the corpus is HCE NDA, where the entire document is unreadable because the embedded font is a subset whose glyph codes are shifted relative to standard encodings and whose /ToUnicode CMap is absent. Several smaller failures in DREU2021 come from symbol fonts whose glyph codes have no Unicode meaning at all.

Work items

  1. In src/pdf/cmap/font_refs.rs, when no /ToUnicode reference is found for a font, fall back to the font's /Encoding entry.
  2. Add encoding tables in a new src/pdf/cmap/encodings.rs for WinAnsiEncoding, MacRomanEncoding, MacExpertEncoding, StandardEncoding, and the dedicated symbolic encodings (Symbol, ZapfDingbats). Tables are public-domain; each is ~256 entries.
  3. Apply the /Differences array on top of the base encoding when present, mapping the named glyphs back to Unicode via the Adobe Glyph List (a compact subset is sufficient — under 500 glyph names cover all observed fonts).
  4. For fonts whose name ends in +Bold, +Italic, +BoldItalic etc., treat the base subset prefix as the encoding hint.
  5. For HCE-NDA-style subset fonts that have neither a /ToUnicode nor a useful /Encoding, detect a constant Caesar-style offset by sampling: take the most common character code in the page, assume it should be U+0020 (space) since prose is mostly spaces, and apply the resulting offset to the rest of the stream. Reject the offset and emit a warning if the recovered text fails a printable-ratio check.
  6. Add a per-page warning when a font's text could not be mapped to Unicode with confidence, so the conversion is honest about the failure rather than emitting cipher text silently.

Files touched

  • src/pdf/cmap/encodings.rs (new)
  • src/pdf/cmap.rs, src/pdf/cmap/font_refs.rs
  • src/pdf/text/strings.rs for the printable-ratio helper

Effectiveness check (must run before declaring Stage 1 done)

  1. Re-run the converter on every file in input/ and diff output/ against the pre-stage snapshot.
  2. For HCE NDA: paragraphs in the article must contain real English words; assert grep -c 'MUTUAL\|CONFIDENTIAL\|HUBBELL\|Inventronics' ≥ 8 (currently 0).
  3. For DREU2021: grep -c '}uW\|}vW\|}v]]}vW' in the article must drop to 0 (currently 28+).
  4. For every other file, the article must stay byte-equal or only gain characters; flag any reduction in grep -c '<p>' as a regression.
  5. Add a tests/conversion_fixtures.rs case that builds a minimal PDF with a Type 1 font using /Encoding /WinAnsiEncoding and no /ToUnicode, asserts the output article contains the correct Unicode text.

Stage 2 — Clean PDF metadata: title source, language, and filename leaks

Four of the ten outputs have a <title> that is either a Word/Excel filename or a one-word watermark. Two documents have working language metadata; the rest have none. This is the cheapest stage in the plan.

Work items

  1. In src/pdf/mod.rs::document_title, reject any /Info /Title value that matches a generated-by-Office pattern: ends in .doc, .docx, .xls, .xlsx, .pdf, .pages, .key, .numbers, or starts with Microsoft Word - , Microsoft PowerPoint - , untitled. Fall back to the next title source.
  2. Read XMP metadata in /Metadata when present and prefer dc:title over /Info /Title.
  3. When no usable title is found, take the first heading-class line of page 1 whose font size is the largest on that page and which is at least three characters long, ignoring single words like "English" or "Draft".
  4. When /Lang is missing, leave <html> without a lang attribute rather than emitting an empty value. Detect "fr-en" or "en-fr" bilingual content if both languages appear in long enough runs and emit two warnings ("document is bilingual; default lang set to X").

Files touched

  • src/pdf/mod.rs
  • src/pdf/streams.rs for XMP stream access

Effectiveness check

  1. grep '<title>' across every output file — no result may include .docx, .xlsx, .pdf, or start with Microsoft Word - .
  2. For IEC 61000-3-2 the title must contain 61000-3-2 and not be the single word "English".
  3. For DREU2021 the title must contain Quotation for INDO Lighting (or fall back to a heading from page 1) but must not contain Microsoft Word.
  4. For IAF-Dimitri the title must not contain .xlsx.
  5. Existing positive cases (XML-Message-for-SCT, Inventronics) must keep their working titles.

Stage 3 — Consolidate multi-line headings and document titles

IEC's cover, the How To Program headings, the Installation Guidelines callouts, and the Inventronics letterhead all emit one <h2> per visual line where the source has one heading wrapped across lines. Consolidation must respect real paragraph boundaries.

Work items

  1. In src/pdf/layout.rs, after the existing heading classifier marks a line as a heading, merge with subsequent lines that:
    • match the same font name and size within ±5%, and
    • begin within one line-height vertically below the previous, and
    • are not themselves a list marker.
    Use a single concatenated string with a single space and emit a single Heading block.
  2. When the merged heading would exceed 200 characters, abort the merge — likely two unrelated headings.
  3. Detect a "stacked title block" on page 1 where ≥ 3 same-font heading lines are vertically contiguous and emit them as one h1 using the largest font.
  4. Demote consecutive identical headings (the IEC pattern where "INTERNATIONAL STANDARD" appears on every cover page) to page-furniture and suppress after the first emission per font / content combination.

Files touched

  • src/pdf/layout.rs
  • src/pdf/postprocess.rs for the duplicate-heading suppression
  • src/pdf/layout_tests.rs for new fixtures

Effectiveness check

  1. For IEC the four-line title cover must produce one <h1> containing "INTERNATIONAL STANDARD" or similar, not four <h2>.
  2. For Installation Guidelines the <h2> count must drop from 3+ to ≤ 1 (one real section heading).
  3. For Inventronics IP Statement the page-1 title must remain a single <h1>.
  4. For XML-Message-for-SCT the per-section <h2> count must not drop (regression guard) — those are real section titles.

Stage 4 — Recover lists from PDF marker glyphs

Inventronics ("1.", "2.", "3."), Installation Guidelines (bullets and arrows), and the protocol docs ("o" sub-bullets) all currently render as flat paragraphs. The existing list detector only handles markers at the start of a clean text line.

Work items

  1. In src/pdf/layout.rs, extend the unordered-marker set with the glyphs typically used in PDFs: •, ·, ▪, ►, ‣, —, o (lowercase Latin "o" used as a sub-bullet in Inventronics-style protocol docs when it's followed by ≥ 2 spaces and the rest of the line is at the same indent across siblings).
  2. For ordered markers, accept "1.", "(1)", "1)", "i.", "I." and uppercase letter prefixes "A.".
  3. When a marker is a tiny separate segment (the Inventronics "1.", "2.", "3." appear as standalone 12pt fragments at the page margin), pair it with the next text-line whose y is within two line-heights and whose x is to the right of the marker.
  4. Detect indented continuation lines (text-line whose x is past the marker's right edge) and append them to the same list-item block instead of starting a new paragraph.
  5. Pre-pass: if the same marker glyph appears at consistent x in ≥ 3 lines within 20 line-heights, treat all those lines as a list rather than as paragraphs.

Stage 5 — Use measured glyph widths for word-boundary classification

The remaining mid-word splits ("Pow er", "default C urrent", "D igitalD im ming", "Rea AC mA)" / "d driver") come from the gap classifier in src/pdf/text/lines.rs using an approximate 0.25 × font_size threshold. The font metrics module already loads per-character widths but only uses them for segment positioning, not for the gap decision.

Work items

  1. In src/pdf/text/lines.rs::push_gap, replace the font-size proportional threshold with the actual width of a space character in the current font, queried from FontMetrics. Fall back to font_size × 0.25 when the font has no recorded space width.
  2. In src/pdf/layout.rs::table_cells the cluster gap threshold should also come from the font's space-width × 2 instead of the constant 21pt fallback.
  3. Add a "joined cell" pass: when two adjacent segments belong to the same font and the gap between them is below a single space width, drop any space that was injected between them in the text (fixes "D igitalD im ming" and "AC mA)" / "d driver").
  4. Cache FontMetrics::space_width per font so the check is O(1) per segment.

  • src/pdf/text/lines.rs

  • src/pdf/layout.rs

Effectiveness check

  1. grep -c 'Pow er\|default C urrent\|D igitalD\|Rea AC mA' across the corpus must reach 0.
  2. For XML-Message-for-SCT: "October 201 0", "January 201 2", and the trailing-digit splits must drop from the article (allowed to remain in the visual reconstruction).
  3. For Digital Dimming Protocol: every command table row should have a single description cell, no mid-word splits.

Stage 6 — Reading order for multi-column and form layouts

IEC's cover is a bilingual two-column layout; the Silica org chart is a free-form positioned slide; IAF-Dimitri is a form where labels and inputs share rows. The current y-then-x sort in src/pdf/text/lines.rs::text_lines walks the page as if it were a single column. The result is interleaved English and French in IEC and shuffled names/titles in Silica.

Work items

  1. Add a column detector in src/pdf/text/lines.rs: scan the segment x positions across the whole page, cluster them into vertical bands, and decide whether the page is multi-column by checking for two or more bands separated by a sustained gap with no segments.
  2. When the page is multi-column, sort within each band before interleaving — i.e. emit all of column 1 top-to-bottom, then column 2, then column 3 — rather than y-major across the whole page.
  3. Detect form layouts: a page where most text is paired with small rectangles or rule lines at consistent x positions. Convert label/value pairs into a <dl> by pairing each label with the value to its right or below.
  4. Use tagged-PDF reading order when /StructTreeRoot provides it, overriding the geometric detector.

Stage 7 — Treat TOC dot leaders as a table of contents, not a table

IEC and XML-Message-for-SCT both have multi-page TOC sections where each entry is "Title …………… 3". The current detector either emits these as paragraphs full of dots or as wide tables with the dots as a middle column.

Work items

  1. In src/pdf/postprocess.rs add a TOC detector that triggers when ≥ 3 consecutive paragraphs each match the pattern ^(.+?)\s*\.{4,}\s*(\d+)$.
  2. Emit those as a <nav class="toc"> containing an <ol> whose items are <li><a href="#section-N">Title</a> <span>page</span></li> (no link target until Stage 8, but the structure is right).
  3. Strip TOC paragraphs from the rest of the article so they don't appear twice.
  4. Reject false positives by requiring the page numbers in the sequence to be monotonically non-decreasing.

Work items

  3. Drop paragraphs whose font size is ≤ 60% of the body font and whose text is a single character or pure digits (footnote anchors).
  4. In src/pdf/visual.rs::render_pages, suppress rectangle shapes whose area is under 4 pt² and which are isolated from any text — these are anti-aliasing artifacts of stroked lines.

Stage 10 — Reduce visual reconstruction noise

The positional view (pdf-recreated-page) is helpful for sighted inspection but currently emits 27 000+ fragments for XML-Message-for-SCT and 8 800 for IEC, many of them adjacent single-word fragments that visibly belong together. Coalescing them makes the visual reconstruction more readable and shrinks the output substantially.

  4. Cap the visual reconstruction at the first N pages (configurable, default 8) so that 100-page documents don't balloon the HTML; emit a hint that the full visual is truncated.

5. Sequencing and stop conditions

Stages 1, 2, 3, 5 and 8 are independent and could land in any order; Stages 4, 6, 7, 9, 10 build on those. The recommended order is the one above, because Stage 1 unlocks meaningful comparison on HCE NDA and DREU2021, and Stage 2 makes the title sample bar usable for every subsequent check.

Each stage's effectiveness check is the gate: if any check fails or any regression guard fires, the stage is not done. Do not move on until the current stage's checks pass.

After all ten stages, the corpus should produce HTML where every document's <title> is meaningful, every paragraph contains real Unicode words, every list is a list, every TOC is a navigation block, every section title is one heading not many, and the visual reconstruction is a fair shadow of the original page rather than a debug dump.
