toHTML Code Review

Review date: 2026-05-22. Reviewer: Claude (Opus 4.7). Scope: implementation under src/ compared against GOAL.md, PDF_READABLE_HTML_SPEC, ISO 32000-1:2008 and ISO 32000-2:2020 (via the spec's clause references), and the sample input/output pairs in input/ and output/.

1. Summary

The architecture and document model match the spec well. The PDF parser is correctly object-graph driven rather than byte-scan, and the text-operator interpreter covers the surface required by Goals 1–5. ToUnicode CMap handling, marked-content roles, ActualText, and Artifact-filtering give the implementation a credible footing for Goals 11–13.

Three things need attention before the project can claim its own Definition of Done:

  1. A <style> block is emitted on every document. This directly violates GOAL.md, the README, and spec §2.3 / Goal 14.
  2. Per-fixture string repairs and a per-fixture synthetic table header have been merged into the core pipeline. These mask Goal 6/7 gaps and will not generalise.
  3. Sample outputs in output/ show regressions the unit tests do not catch: corrupted lang, missing PDF titles, no PageBreak markers, paragraph fragments classified as headings, and broken word boundaries that the Stage 9 note already identified as open.

3. Critical Issues

3.1 The renderer always emits CSS

src/html/mod.rs::render_default_styles writes an inline <style> block on every document. This contradicts:

  • GOAL.md: "The generated document must not include CSS or JavaScript. Structure should come from HTML elements and attributes only."
  • README.md: "No CSS or JavaScript output."
  • Spec §2.3: "no CSS and no JavaScript by default."
  • Spec Goal 14: "The converter MUST NOT use CSS positioning to recreate the PDF page."
  • STAGE_9_PDF_SEMANTICS: "Output font and pixel fidelity are intentionally out of scope."

The accompanying test at src/html/mod.rs:110 is named renders_complete_html_document_without_css_or_javascript but asserts that font-size: 12px is present. The test name and its assertion are in direct conflict and should be inverted.

Recommended fix: delete render_default_styles and its caller, then change the test to assert !html.contains("<style").

3.2 Per-fixture string repairs in core code

src/pdf/postprocess.rs:83 defines a REPAIRS array of more than fifty hard-coded string substitutions such as "Annex BClieop03" → "Annex B Clieop03" and "Version7.0tFebruary2013" → "Version 7.0 - February 2013". These are tuned to the XML-Message-for-SCT sample. They do not generalise, will rot quickly, and disguise the real Goal 6/7 gap: font-width-based gap inference should be putting the missing spaces in.

Similarly, src/pdf/layout.rs:329 hosts a message_item_header / synthetic_message_item_header pair whose only purpose is to label that one PDF's columns "MessageItem", "<XMLTag>", "Mult.", "Represent./Type". This is per-document logic in a general-purpose converter.

Recommended fix: move both to a dedicated test fixture, then replace them with a real gap classifier in layout.rs that uses FontMetrics::text_width plus a body-text font-size floor to decide when adjacent segments should join or insert a space.

3.3 PDF document title and language are not extracted properly

Every converted PDF in output/ shows <title>toHTML document</title>. DocumentMetadata.title is never set from the PDF's /Info /Title entry or XMP metadata, even though the spec §5 maps that explicitly to DocumentMetadata.title.

src/pdf/mod.rs::document_language finds the bytes after /Lang with a substring search and reads everything between the first ( and ) as a Latin-1 string. It does not handle hex strings, indirect references, or UTF-16BE literals with a BOM. The Inventronics output saves lang="�� E N - U S" — the 0xFE 0xFF BOM has been emitted as replacement characters and the UTF-16 bytes have been read as individual characters.

Recommended fix: read /Info via PdfObjects, decode each string with the same routine as text-content strings (covering hex, UTF-16BE, and the PDFDocEncoding default), and reuse that routine for /Lang.

3.4 No PageBreak between pages

Block::PageBreak exists in the model but src/pdf/mod.rs::pdf_to_document never emits one. Spec §4.4 lists this as SHOULD, and §5 maps page boundaries to Block::PageBreak. The HTML renderer already supports rendering it (src/html/blocks/pages.rs).

Recommended fix: after the inner loop in pdf_to_document, append a Block::PageBreak carrying page.page_number when more than one page exists and the page produced any blocks.

3.5 Heading classifier is too eager

src/pdf/layout.rs::heading_line calls a line a heading whenever its font size is at least 1.25× the line median. On the Installation Guidelines sample this produces eleven <h2> elements, many of which are mid-paragraph fragments such as "Best", "Avoid Vertical mounting, enabling water to collect at junction", and full sentences ending in periods that nonetheless beat the median by enough to qualify.

Recommended fix: require additional evidence before classifying a line as a heading: short length, no terminal period (already checked), isolated vertical spacing both above and below, and either a tagged role or a bold-flagged font.

4. Repository-Wide Quality Observations

  • Module decomposition is good. src/pdf/text/ and src/pdf/cmap/ are broken into reader, state, parser, strings, syntax, and tests. The Stage 9 note's worry that PDF parsing could become a hotspot has been actively managed.
  • The document model in src/model.rs matches the spec §5 mapping table. Every block type the spec calls out has a corresponding variant; SourceSpan carries the page number the spec asks for.
  • render_html is the right shape — warnings are surfaced in a discoverable <section data-conversion-warnings>, page placeholders carry data-page and data-reason, and the title is escaped. Removing the inline stylesheet is the only change required for it to match the spec.
  • The CLI exists (src/cli.rs with an interactive sub-module), covering GOAL.md Stage 8.
  • Fixtures need to grow. The Stage 16 / spec Goal 16 checklist is far ahead of the current tests/conversion_fixtures.rs. The Stage 9 note is honest about this; the work simply has not been done yet.

5.1 How To Program a Driver

Input → output.

  • Selectable headings are recovered.
  • The detected URL is wrapped in <a> correctly.
  • Spec-sheet tables are image XObjects — correctly emitted as a warning.
  • Word boundaries are visibly broken (withoutChangingthe, Pow er, Ichange, programm er). These are the same strings Stage 9 already lists as open and they are the most visible reason to finish Goal 7 properly rather than expanding REPAIRS.

5.2 Inventronics IP Statement

  • lang="�� E N - U S" — see §3.3.
  • Title heading lost word breaks (RE:PHILIPSPATENT CLAIMS) — same Goal 7 issue.
  • Paragraphs are clipped mid-sentence ("…of others .").
  • Inline <style> block present — see §3.1.

5.3 Installation Guidelines — Moisture Ingress

Input → output.

  • 11 <h2> elements — most are paragraph fragments. See §3.5.
  • Bullets are emitted as <p>; list-marker detection misses the glyphs used here.
  • One table is detected, but its header reads "Through - h o les on top cove r of fixture" — another Goal 7 word-gap failure.

5.4 XML Message for SCT v7

Input → output.

  • 530 paragraphs, 0 <h1>. The document title is split across five <h2> tags. The PDF's /Info /Title never reaches the HTML title.
  • Table reconstruction on the Change History table is correct, including right-alignment for numeric cells.
  • The XML appendix is correctly collapsed to <pre><code> by postprocess::collapse_code_blocks. This is the cleanest piece of the post-processing layer.
  • Many TOC dot-leader lines survive as paragraphs ("Introduction.......3"). A TOC detector could either drop these or render them into a proper list.

6. Recommended Next Steps

  1. Delete render_default_styles and invert the misleading test in src/html/mod.rs. < 1 hour.
  2. Read /Info /Title and decode UTF-16BE / hex strings; reuse the decoder for /Lang. 1–2 hours.
  3. Emit Block::PageBreak between pages in src/pdf/mod.rs::pdf_to_document. < 1 hour.
  4. Replace the REPAIRS table and synthetic_message_item_header shim with a real gap classifier in src/pdf/layout.rs driven by FontMetrics::text_width. 1–2 days.
  5. Tighten the heading classifier with isolation, bold-flag, and tagged-role evidence.
  6. Walk /StructTreeRoot for tagged-PDF support beyond marked-content roles.
  7. Grow fixtures for incremental updates, multi-byte CMaps, ActualText spans, image-only pages, and repeated header/footer detection.

7. Closing

The bones of a faithful PDF-to-HTML converter are in place. The most impactful single change is the smallest one: stop emitting the inline stylesheet. After that, the priority is finishing Goal 7 properly so that the REPAIRS table can be deleted, followed by real PDF metadata handling so titles and languages survive.
